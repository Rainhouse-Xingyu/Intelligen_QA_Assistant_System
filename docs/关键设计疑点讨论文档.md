## 疑点一：微信公众号的角色定位

### 现状分析

**需求文档怎么写**（考务 4）：
> "可集成至教务部现有公众号平台"

**任务分配表怎么写**（任务 2）：
> "微信公众号平台集成 — 微信公众平台接口 + Java SpringBoot"
> 前端任务："协助测试多端（手机端/公众号内）的页面样式兼容性"
> 后端任务："负责教务部现有公众号平台的后台对接与消息路由"

**数据库设计**：
> `sys_user.wechat_openid` — 微信公众号 OpenID 字段（用于 OAuth 登录关联）

**API 设计**：
> `GET /api/wechat/login` — 微信 OAuth 回调，传入 code 换取系统 JWT Token
> `POST /api/wechat/portal` — 接收微信公众号用户的消息或推送事件（关注、取消等），返回自动响应 XML

### 核心疑点

微信公众号在本系统中到底承担什么角色？存在两种理解路径：

| | 方案 A：纯渠道入口 | 方案 B：完整交互载体 |
|---|---|---|
| **定位** | 公众号仅作为引流入口 + 通知推送渠道 | 师生可直接在公众号聊天窗口内完成全部 AI 对话 |
| **交互方式** | 点击菜单 → 跳转 H5 网页 → 在网页内对话 | 在公众号对话框直接发文本/语音，AI 通过公众号消息接口回复 |
| **技术实现** | 公众号菜单 → OAuth 授权 → 内嵌 WebView 加载 Vue 前端 | `POST /api/wechat/portal` 接收 XML 消息 → 转发 Coze → 拼接 XML 回复 |
| **前端工作量** | 需适配移动端 H5 响应式（已在规划中） | 无需前端页面，纯后端 XML 消息处理 |
| **优势** | 功能完整（语音、心理倾诉、资源推荐等全功能可用），UI 可控 | 零前端开发，用户无需跳出微信 |
| **劣势** | 需跳出微信或加载 WebView，体验有割裂感 | 微信消息接口限制多（5 秒超时、无法发语音、无法做复杂 UI、无法做多轮对话状态管理） |
| **数据库支撑** | `wechat_openid` 做身份关联即可 | 需额外存储微信 `openid` ↔ 会话映射，被动回复有 5 秒超时限制 |

### 关键约束

微信公众平台消息接口的硬限制：
- **被动回复 5 秒超时**：用户发消息后 5 秒内必须回复，否则微信断开连接。Coze 工作流响应时间通常远超 5 秒。
- **客服消息接口**：48 小时内可主动推送，但需要额外开发。
- **语音消息**：微信可接收语音消息并传给服务器，但格式为 `.amr`，需额外转码。
- **无法承载复杂 UI**：词云、图表、时间轴、资源卡片等在纯文本聊天窗口无法呈现。

### 建议

**推荐方案 A（渠道入口）**，理由：
1. 微信 5 秒超时限制与 Coze AI 响应时间（通常 3-15 秒）不兼容
2. 系统需求包含语音播放、资源卡片、时间轴、心理倾诉等多模态交互，无法在纯文本聊天窗口实现
3. 公众号菜单 + OAuth + 内嵌 H5 是高校信息化系统的主流集成模式

**公众号应承担的功能**：
| 功能 | 说明 |
|------|------|
| 身份认证入口 | 微信 OAuth 一键登录，免去输入账号密码 |
| 菜单导航 | "智能问答"/"心理倾诉"/"学业档案"等快捷入口，点击跳转对应 H5 页面 |
| 模板消息推送 | 学业预警通知、帮扶方案更新提醒、热门问题推送（任务 10） |
| 关注/取消关注事件 | `POST /api/wechat/portal` 已实现，用于用户生命周期管理 |

### 待与老师确认

- [ ] 公众号中是否需要支持"直接在聊天框对话"？（如需要，必须解决 5 秒超时问题，建议用客服消息异步回复）
- [ ] 公众号菜单结构初步期望？（建议：智能问答 / 心理倾诉 / 我的档案 / 更多）
- [ ] 是否需要公众号模板消息推送（预警通知等）？

## 疑点二：语音消息的完整处理链路

### 现状分析

**需求文档怎么写**（教学 3）：
> "师生可通过语音输入或文字输入进行提问，智能问答助手能够自动识别问题意图，在问答库中进行匹配，并分别通过语音和文字两种方式给出解答反馈"

**任务分配表怎么写**（任务 4）：
> "在 Coze 中配置语音双向转化节点（ASR 语音识别 + TTS 语音合成），前端编写录音与播放组件"

**API 设计**：
> `POST /api/chat/voice` — 发送语音消息（入参: `audioFile` MultipartFile）
> "返回值: AI 对该语音内容的**文字类型**答复" — 注意当前 API 返回 `Result<String>` 文本

**数据库设计**：
> `chat_message.msg_type` = `2`（语音），`media_url` 存储语音文件链接
> `chat_message.sender_type` = `1`(用户) / `2`(AI) / `3`(人工客服)

### 核心疑点

```
用户录音 ──→ ?? ──→ ?? ──→ AI处理 ──→ ?? ──→ 返回给用户
             ①         ②                      ③
```

**疑点 ①: 语音传到哪里？**

| 方案 | 流程 | 优劣 |
|------|------|------|
| **A: 语音→后端→Coze** | 前端录音 → `POST /api/chat/voice` 上传音频文件 → 后端转发 Coze → Coze 内 ASR 插件转文字 → LLM 处理 → TTS 合成语音 → 返回 | ✅ 保留原始语音，Coze 内置 ASR/TTS 插件 ✅ 任务表明确写"Coze 中配置语音双向转化节点" |
| **B: 前端 ASR** | 浏览器 `MediaRecorder` 录音 → 浏览器端 Web Speech API / 第三方 ASR 转文字 → 文字发 `POST /api/chat/text` | ❌ 丢失原始语音 ❌ 浏览器 ASR 准确率低 ❌ 与任务表"Coze 插件"方案冲突 |

**建议选方案 A**：录音文件完整上传后端 → 转发 Coze → Coze 内 ASR→LLM→TTS 全链路处理。

**疑点 ②: 语音中的情绪/语气需要保留吗？**

需求文档提到心理倾诉功能（学业 3："利用 AI 助手给学生提供一个可交流倾诉的对象"），这涉及一个深层问题：

语音中包含的**情绪特征**（语速、音调、停顿、颤抖等）对心理咨询场景有价值，但 Coze 的 ASR 插件通常只输出纯文本，会丢失副语言信息。

| 场景 | 处理方式建议 |
|------|-------------|
| **常规考务/学业问答** | ASR→文字→Coze 工作流→文字回复 + 可选 TTS 朗读。情绪不重要，准确识别问题内容即可。 |
| **心理倾诉场景** | 保留原始音频（`media_url` 已经存储），AI 基于 ASR 文字做共情回复。若风险等级较高（自伤倾向等），系统标记并通知辅导员人工介入，原始语音可供辅导员回听。 |
| **转人工时** | 当前转人工 API `POST /api/chat/transfer-to-human` 不传媒体。建议转人工时将原始语音文件 `media_url` 一并发送给客服端播放。 |

**疑点 ③: AI 回答是否需要语音合成？**

| 方案 | 说明 |
|------|------|
| **仅返回文字** | 当前 API 返回 `Result<String>` 文本，前端直接渲染文字气泡 |
| **文字 + 语音双返回** | API 返回 `{ "text": "...", "audio_url": "..." }`，前端同时展示文字 + 音频播放条 |
| **用户可选** | 每个回答旁加"播放语音"按钮，按需触发 TTS |

需求文档明确写"分别通过语音和文字两种方式给出解答反馈"，任务表中也有"Coze 语音合成 TTS 节点" — **建议支持双模态返回**。

### 语音消息完整链路建议

```
┌─────────────────────────────────────────────────────┐
│ 前端                                                 │
│  录音 → Blob → POST /api/chat/voice                  │
│  接收 { text: "...", audio_url: "..." }              │
│  展示文字气泡 + 语音播放条                            │
└──────────────────┬──────────────────────────────────┘
                   │ audioFile (MultipartFile)
┌──────────────────▼──────────────────────────────────┐
│ Java 后端                                            │
│  保存音频 → media_url                                │
│  写入 chat_message (msg_type=2，sender_type=1)       │
│  转发 Coze 工作流                                    │
└──────────────────┬──────────────────────────────────┘
                   │
┌──────────────────▼──────────────────────────────────┐
│ Coze 工作流                                          │
│  ASR 插件（语音→文字）                                │
│  → LLM 问答（匹配知识库/心理倾诉Agent）               │
│  → TTS 插件（文字→语音，可选）                        │
│  → 返回 { text, audio_url }                          │
└──────────────────┬──────────────────────────────────┘
                   │
┌──────────────────▼──────────────────────────────────┐
│ 未识别兜底 → 用户点"转人工"                           │
│  后端将原始 media_url 附带转发给人工客服台            │
│  客服台播放用户原始录音 + 查看 ASR 文字               │
└─────────────────────────────────────────────────────┘
```

### 待与老师确认

- [ ] 语音处理方案：是否采用"前端录音→后端中转→Coze ASR+LLM+TTS"全链路？
- [ ] 心理倾诉场景是否需要保留原始语音的情绪特征供辅导员回听？
- [ ] AI 回答是否必须返回语音合成（TTS），还是仅返回文字即可？
- [ ] 转人工时是否需要将用户的原始语音文件一并传给客服？当前 API 不支持，需扩展。

## 疑点三：知识图谱的形态与课程同步

### 现状分析

**需求文档怎么写**（学业 2）：
> "结合知识图谱，根据不同学生情况推送适配性、个性化学习资源"

**当前数据库**：
> 无课程表、无知识图谱相关表。
> `student_profile` 仅有 `gpa` 和 `failed_courses_cnt`，**无分科成绩、无当前学期选课记录**。

### "知识图谱" 在此系统中的可能含义

"知识图谱"易与通用知识图谱（如 Wikidata）混淆。在此教育场景下，更可能指**课程依赖关系图**：

```
示例：高等数学(上) ──→ 高等数学(下) ──→ 概率论
                    └──→ 线性代数 ──→ 机器学习
                    
学生 A：高等数学(上)挂科 → 红色标记 → 影响下游 3 门课 → 推荐先重修
学生 B：全部通过 → 绿色标记 → 推荐进阶选修
```

### 核心疑点

**疑点 ①: 知识图谱以什么前端形式展现？**

| 展现形式 | 适用场景 | 技术方案 |
|----------|----------|----------|
| **节点-连线图**（力导向图） | 展示课程依赖全貌、学生当前所处位置标记 | ECharts 关系图 / D3.js 力导向图 |
| **树状层级图** | 展示某一专业方向的课程递进关系 | ECharts 树图 |
| **学生个人课程地图** | 展示该生已修/在修/未修课程的完成状态（红黄绿标识） | 定制化课程卡片网格 + 连线标注依赖 |
| **管理端全局图谱** | 教务老师查看所有课程依赖关系，编辑/调整 | 交互式力导向图 + 拖拽编辑 |

**建议**：
- **管理端**：全局力导向图（可编辑节点和边）
- **学生端**：个人课程地图卡片（已完成/进行中/未解锁 + 风险提示）

**疑点 ②: 是否需要同步学生本学期课程信息？**

| 方案 | 说明 | 数据来源 |
|------|------|----------|
| **A: 不需要同步** | 仅使用 `student_profile` 中的汇总 GPA + 挂科数，不做单科分析 | 现有 DB 即可 |
| **B: 手动导入** | 教务老师通过"学生画像管理"页面批量上传 Excel（含本学期选课列表） | 手动维护 |
| **C: 对接教务系统** | 从学校教务系统 API 同步学生选课数据、成绩单、课程表 | 需要学校提供对接接口 |

**需求文档的判断**：

需求文档学业1 说"可分析学生学业指标以及学业情况调查问卷的内容" —
这意味着至少有**成绩导入**和**调查问卷数据**两个数据源。但需求文档没有明确说是否需要对接教务系统实时同步。

任务 12 说"提供学生学业指标、调查问卷数据的本地存储与调用接口" — 数据存储在本地 DB。

**目前 DB 层面的缺口**：

要支持"知识图谱 + 个性化课程信息"，当前 `llm_qa_sy.sql` 缺少以下表：

| 缺失的表 | 用途 |
|----------|------|
| `course` | 课程基础信息（课程编号、名称、学分、学期、先修课程 ID） |
| `student_course` | 学生选课关系（学生 ID、课程 ID、学期、成绩、是否补考/重修） |
| `course_prerequisite` | 课程先修依赖关系（课程 ID → 先修课程 ID），构成知识图谱的"边" |
| `survey_response` | 学业情况调查问卷作答记录（问卷 ID、学生 ID、各项指标得分） |

### 建议

1. **知识图谱前端展现**：管理端用 ECharts 力导向图展示课程依赖全貌；学生端展示个人课程完成卡片地图
2. **课程信息同步**：建议**方案 B（手动导入）作为 MVP**，方案 C（对接教务系统）作为远期目标
3. **数据库补齐**：至少增加 `course` 和 `student_course` 两张表以支撑知识图谱的基础数据

### 待与老师确认

- [ ] "知识图谱"具体指什么：课程依赖关系图 / 知识点图谱 / 学生学习画像网络？
- [ ] 课程信息是否需从学校教务系统实时同步？同步频率（每学期/每日）？
- [ ] MVP 阶段是否接受手动 Excel 导入课程数据？
- [ ] 知识图谱是否需要在前端交互编辑（增删节点/连线），还是仅只读展示？
- [ ] 问卷数据来源：学校现有的学业调查问卷系统，还是系统内新建电子问卷？

## 疑点四：个性化学习资源的来源与存储

### 现状分析

**需求文档怎么写**（学业 2）：
> "结合知识图谱，根据不同学生情况推送适配性、个性化学习资源"

**任务分配表怎么写**（任务 13）：
> "在 Coze 内配置资源匹配逻辑"
> 后端任务："在后端建立学生画像标签系统，为 AI 提供脱敏后的学生特征"

**当前数据库**：
> **没有任何资源相关的表。**
> `student_profile` 有 `psychological_tag`、`risk_level` 标签字段（可用于匹配资源）
> `student_growth_archive` 有 `help_plan`（AI 生成的帮扶方案文本，但不等于资源库）

### 个性化推荐的完整链路

```
学生画像（GPA/挂科/心理标签/风险等级）
         │
         ▼
    Coze Agent（资源匹配逻辑）
         │
         ▼
    推荐资源列表 ──→ 返回前端展示
         │
         △
    ┌────┴────┐
    │ 资源从哪里来？ ← 核心问题
    └─────────┘
```

### 核心疑点：资源存储在哪里？

| 方案 | 说明 | 优劣 |
|------|------|------|
| **A: Coze 知识库** | 将学习资料（课件 PDF、视频链接、复习指南等）上传到 Coze 知识库，Agent 在知识库中检索匹配 | ✅ 零运维成本 ✅ Coze 内置向量检索 ❌ 存储空间有限 ❌ 文件管理不便 ❌ 更新需手动同步 |
| **B: 学校自建资源服务器** | 学校有独立的教学资源服务器（如 Moodle、Blackboard、自建 OSS），提供 API 供系统调用 | ✅ 与学校现有系统打通 ✅ 存储无限制 ✅ 资源可被多系统复用 ❌ 需要学校 IT 配合提供 API ❌ 需要对接开发 |
| **C: 系统内置资源表** | 在 `llm_qa_sy` 库中新建资源相关表（`resource`、`resource_tag`），管理员通过后台录入/上传，存储到本地 OSS | ✅ 完全自主可控 ✅ 可与画像标签联动 ❌ 需额外开发资源管理后台 ❌ 需自行维护存储 |
| **D: 混合方案** | 资源元数据（标题、摘要、类型、标签、适用画像条件）存储在本地 DB，实际文件存储在 Coze 或 OSS；Coze Agent 只做匹配决策，返回资源 ID 列表，前端通过本地 API 拉取资源详情 | ✅ 兼顾灵活性和可控性 ✅ 匹配逻辑与资源存储解耦 |

### 建议

**推荐方案 D（混合方案）**，理由：
1. 资源元数据管理需要 CRUD 后台界面对接教务老师，不适合放在 Coze 里操作
2. Coze Agent 擅长"根据学生标签做匹配决策"，不擅长"管理资源库"
3. 资源文件本身可能较大（视频、课件 PDF），需独立存储

**建议的数据表设计**（待补充到 `llm_qa_sy.sql`）：

```sql
-- 学习资源表
CREATE TABLE `learning_resource` (
  `id` bigint PRIMARY KEY AUTO_INCREMENT,
  `title` varchar(200) NOT NULL COMMENT '资源标题',
  `description` text COMMENT '资源描述/摘要',
  `resource_type` tinyint COMMENT '类型: 1-文档, 2-视频, 3-音频, 4-链接, 5-活动',
  `file_url` varchar(255) COMMENT '文件/链接地址',
  `tags` varchar(500) COMMENT '标签(逗号分隔): 如"重修,高数,心理调适"',
  `target_risk_level` tinyint COMMENT '适用风险等级(0-3, NULL=全部)',
  `target_tags` varchar(500) COMMENT '适用画像标签(匹配 student_profile.psychological_tag)',
  `status` tinyint DEFAULT 1 COMMENT '0-下架, 1-上架',
  `created_by` bigint COMMENT '上传管理员ID',
  `created_at` datetime,
  PRIMARY KEY (`id`)
);
```

**数据流**：
```
Coze Agent收到学生画像标签 (risk_level=2, psychological_tag="焦虑")
  → 匹配逻辑返回: "适合资源 ID=5(减压音频), ID=12(学习方法指南)"
  → 后端查 learning_resource 表取元数据
  → 前端渲染资源卡片
  → 学生点击 → 访问 file_url
```

### 待与老师确认

- [ ] 学校目前是否有统一的教学资源平台（Moodle / Blackboard / 自建 OSS）可以利用？
- [ ] 学习资源谁提供、谁审核、谁更新？教务老师 / 辅导员 / 各学院教学秘书？
- [ ] MVP 阶段资源量预估（100 条 / 500 条 / 1000+）？
- [ ] 资源类型范围：是否包含外部链接（B站视频、MOOC课程等），还是仅限校内资源？
- [ ] 推荐匹配逻辑是否需要"过度推荐控制"（如同一学生一周内最多推送 5 条，避免信息过载）？

| 优先级 | 疑点 | 影响范围 | 建议方向 |
